## Overview

This document defines the primary user journeys through LifeOS. Each journey is written from the operator's perspective, describes the steps they take, and identifies the modules, agents, and outputs involved.

Journeys are designed around the principle: **every interaction should leave the system more useful than before.**

---

## Journey 1: First-Time Setup

**Actor:** New operator  
**Goal:** Transform a blank LifeOS installation into a functional operating system  
**Duration:** 30–90 minutes  
**Outcome:** LifeOS is ready to manage at least one business, with AI agents configured and one MCP connector active

### Steps

**Step 1 — Onboard to the Platform**
- Read `README.md` and `docs/REPOSITORY_GUIDE.md`
- Review the architecture overview in `docs/ARCHITECTURE.md`
- Understand the naming conventions in `docs/NAMING_CONVENTIONS.md`

**Step 2 — Define Your First Business**
- Copy the business template from `businesses/_templates/business/`
- Create `businesses/[your-business-slug]/`
- Fill in: `README.md`, `vision/README.md`, `strategy/README.md`
- Assign yourself as owner; assign Chief of Staff as AI owner

**Step 3 — Configure Your First Agent**
- Open `AI/Agents/chief-of-staff/`
- Review `MISSION.md` and `PROMPTS.md`
- Customize the system prompt in `PROMPTS.md` with your business context
- Review `INITIALIZATION.md` and run the initialization checklist

**Step 4 — Connect Your First MCP Tool**
- Choose your most-used external tool (recommend: GitHub or Slack)
- Open `MCP/[connector]/`
- Follow `AUTH.md` to configure authentication
- Verify connection via `STATUS.md` health check

**Step 5 — Create Your First Project**
- Copy the project template from `projects/_templates/project.md`
- Create `projects/[business-slug]/[project-slug]/`
- Fill in all required fields (status, owner, next action, review date)
- Link to your business

**Step 6 — Configure the Command Center**
- Review `command-center/README.md`
- Set your daily review time
- Configure the Chief of Staff agent to generate a daily briefing

**Step 7 — Verify Setup**
- All required fields in your first business template are filled
- At least one project exists with a next action
- Chief of Staff agent is configured
- At least one MCP connector is active
- Command Center is generating a daily briefing

## Journey 2: Daily Execution

**Actor:** Active operator  
**Goal:** Start the day with clarity, execute the highest-value work, and end with everything updated  
**Duration:** 10–15 minutes for review; remainder for focused work  
**Outcome:** Top priorities actioned, decisions logged, system updated

### Morning

**Step 1 — Open Command Center**
- Chief of Staff agent generates the daily briefing
- Review: Today's Priorities, Decision Queue, Waiting On, Risks, Agent Tasks

**Step 2 — Review Priority List**
- AI surfaces the 3–5 highest-ROI tasks across all businesses
- Operator confirms, reorders, or overrides priorities
- Each priority links to its source (project, business, or escalation)

**Step 3 — Check Decision Queue**
- Review decisions awaiting operator input
- For each: select from options, record rationale, finalize
- Decisions auto-update linked projects and knowledge objects

**Step 4 — Review Agent Activity (Previous 24 Hours)**
- Check AI Workspace dashboard: what did agents complete?
- Review any new knowledge objects created
- Approve or modify agent outputs that require human review

### During the Day

**Step 5 — Execute Work**
- Work directly within LifeOS-managed projects and businesses
- Agent tasks run in the background (research, code review, status updates)
- Capture decisions, risks, and insights immediately as they occur — not at end of day

**Step 6 — Log in Real Time**
- Any new decision → Decision Center
- Any new risk → project risk register
- Any new insight → Knowledge Engine (quick capture)
- Any new task → project next action

### End of Day

**Step 7 — Update Project Status**
- Project Manager agent scans for stale next actions
- Operator updates any projects touched during the day
- Confirm or reassign any tomorrow-critical items

**Step 8 — Close the Day**
- Command Center marks today's top priorities as done/deferred/blocked
- Tomorrow's priority list is pre-generated by Chief of Staff
- Any outstanding escalations are acknowledged

## Journey 3: Weekly Review

**Actor:** Active operator  
**Goal:** Review the full system health for the week, identify what changed, and reset priorities for the next week  
**Duration:** 45–90 minutes  
**Outcome:** Updated project statuses, captured learnings, clean decision queue, reset weekly priorities

### Steps

**Step 1 — Trigger Weekly Review**
- Open `command-center/weekly/` and start the weekly review template
- Chief of Staff agent pre-populates the review with data from all modules

**Step 2 — Review Business Health**
- For each active business: check KPI dashboard
- Flag businesses trending negatively
- Note any businesses with no activity this week

**Step 3 — Review Project Portfolio**
- Project Manager agent generates project health summary
- Identify: stalled projects, overdue review dates, unresolved blockers, missing next actions
- Update or archive any inactive projects

**Step 4 — Review Agent Performance**
- AI Workspace: tasks completed, escalation rate, errors
- Identify underperforming agents; adjust prompts or memory configuration
- Review new knowledge objects created this week

**Step 5 — Review Decisions Made This Week**
- Decision Center: decisions logged, pending decisions cleared
- Check for any decisions that need follow-up

**Step 6 — Review Automations**
- Automation Status dashboard: run counts, error rates, failures
- Fix or disable broken automations
- Identify new automation opportunities

**Step 7 — Capture Weekly Learnings**
- Write 3–5 atomic knowledge objects from this week's work
- Link knowledge objects to relevant projects and businesses

**Step 8 — Set Next Week's Priorities**
- Define the 3 most important outcomes for next week per business
- Assign AI agents to appropriate prep tasks
- Confirm review dates for any projects due next week

**Step 9 — Commit the Review**
- Save the completed weekly review to `command-center/weekly/YYYY-MM-DD.md`
- Chief of Staff agent distributes summary to configured channels

## Journey 4: Monthly Review

**Actor:** Active operator  
**Goal:** Assess business health at the monthly level, evaluate strategy, and update the roadmap  
**Duration:** 2–4 hours  
**Outcome:** Updated business KPIs, strategy review complete, roadmap adjusted, monthly summary documented

### Steps

**Step 1 — Trigger Monthly Review**
- Open `command-center/monthly/` and start the monthly review template
- Chief of Staff agent pre-populates all data sections

**Step 2 — Financial Review (Per Business)**
- Finance Hub: MRR, ARR, burn, runway, gross margin
- Flag anomalies; identify trends
- Compare against monthly targets

**Step 3 — Strategy Review (Per Business)**
- Review `businesses/[name]/strategy/README.md`
- Are the 90-day priorities still correct?
- Has the competitive landscape changed?
- Update as needed

**Step 4 — Project Portfolio Review**
- What was completed this month?
- What is on track, at risk, or stalled?
- What should be started, stopped, or accelerated?

**Step 5 — Knowledge Review**
- Knowledge Engine: new objects created, total count, domain coverage
- Identify knowledge gaps that are actively hurting execution

**Step 6 — People Review**
- Review team capacity and contribution
- Identify anyone overwhelmed or underutilized

**Step 7 — AI & Automation Review**
- What tasks have been automated this month?
- What is the estimated time saved?
- Which agents are underutilized?

**Step 8 — Set Monthly Targets**
- Define measurable targets for next month per business
- Update KPI dashboard targets
- Set one stretch goal per business

**Step 9 — Commit the Review**
- Save to `command-center/monthly/YYYY-MM.md`
- Chief of Staff agent distributes to relevant stakeholders

## Journey 5: Creating a Business

**Actor:** Operator adding a new business to LifeOS  
**Goal:** Stand up a fully structured new business that shares the platform  
**Duration:** 1–2 hours  
**Outcome:** New business with all sub-modules configured, linked to platform services

### Steps

1. Copy `businesses/_templates/business/` → `businesses/[new-business-slug]/`
2. Fill in `README.md` (name, status, owner, model, quick links)
3. Write `vision/README.md` (mission, vision, why, constraints)
4. Write `strategy/README.md` (priorities, model, 90-day plan)
5. Configure `ai/README.md` (assign agents, custom prompt context)
6. Configure `kpis/README.md` (define KPIs, set baselines)
7. Assign or create a GitHub organization in `github/README.md`
8. Link relevant MCP connectors in `ai/README.md`
9. Open Decision Center and log the decision to create this business
10. Create the first project for this business

## Journey 6: Creating a Project

**Actor:** Operator starting a new initiative  
**Goal:** Create a fully specified project ready for execution  
**Duration:** 15–30 minutes  
**Outcome:** Project with owner, next action, AI agent, GitHub link, review date

### Steps

1. Copy `projects/_templates/project.md` → `projects/[business]/[project-slug]/project.md`
2. Fill all required fields: title, business, status (Draft), priority, owner, AI owner, review date
3. Write the summary: what is this, why does it matter, what is success?
4. Define the first next action (single, specific, owned)
5. List known deliverables with owners and due dates
6. Identify and document known dependencies
7. Link the GitHub repository (create if needed)
8. Assign MCP connections needed for this project
9. Create the initial risk register (at least one risk entry)
10. Set the first review date (no more than 14 days out)
11. Notify the AI project manager agent

## Journey 7: Connecting GitHub

**Actor:** Technical operator adding a repository to LifeOS  
**Goal:** Full bidirectional connection between a GitHub repo and a LifeOS project  
**Duration:** 15–30 minutes  
**Outcome:** GitHub activity visible in LifeOS; LifeOS project updates linked to GitHub issues

### Steps

1. Open `MCP/github/AUTH.md` and configure the GitHub connector
2. Verify connection via `MCP/github/STATUS.md`
3. Open the relevant project in `projects/`
4. Add the GitHub repo URL to the `github_repo` field
5. Open `github/repositories.md` and register the repo
6. Configure relevant GitHub Actions from `github/actions/`
7. Set up the Engineering Agent to monitor this repo
8. Enable the Repository Health dashboard widget for this repo
9. Create the first linked issue in GitHub from a project deliverable

## Journey 8: Connecting MCP Tools

**Actor:** Operator adding an integration  
**Goal:** Add a new MCP connector to extend LifeOS capabilities  
**Duration:** 10–20 minutes per connector  
**Outcome:** New connector active, authenticated, and available to agents and automations

### Steps

1. Identify the connector needed (see `MCP/` directory)
2. Open `MCP/[connector]/README.md` — review capabilities and use cases
3. Follow `MCP/[connector]/AUTH.md` — configure authentication (API key, OAuth, etc.)
4. Review `MCP/[connector]/CAPABILITIES.md` — understand what the connector can do
5. Review `MCP/[connector]/DEPENDENCIES.md` — check for prerequisites
6. Verify the connection by checking `MCP/[connector]/STATUS.md`
7. Open `AI/Agents/[relevant-agent]/TOOLS.md` and add the connector
8. Test with a simple agent task that uses the new connector
9. Create an automation in Automation Studio that leverages the connector

## Journey 9: Creating an AI Agent

**Actor:** Operator building a new specialized agent  
**Goal:** Define, configure, and deploy a new AI agent  
**Duration:** 30–60 minutes  
**Outcome:** Agent with full configuration, tested and ready for live tasks

### Steps

1. Determine the agent's role: what unique job will this agent do?
2. Copy `AI/Agents/_templates/` → `AI/Agents/[agent-slug]/`
3. Write `MISSION.md`: purpose, who it serves, success criteria
4. Define `TOOLS.md`: which MCP connectors, which capabilities
5. Write `PROMPTS.md`: system prompt with role, behavior, constraints, output format
6. Configure `MEMORY.md`: short-term window, long-term knowledge sources, injected context
7. Define `WORKFLOWS.md`: which workflows trigger this agent
8. Set `METRICS.md`: what to measure, what thresholds trigger review
9. Complete `INITIALIZATION.md`: step-by-step setup instructions
10. Test with 3–5 representative tasks
11. Register the agent in the Agent Registry (`AI/Agents/README.md`)
12. Update `README.md` with validated performance baseline

## Journey 10: Capturing Knowledge

**Actor:** Operator or AI agent capturing a validated insight  
**Goal:** Create an atomic knowledge object that is AI-consumable and linked  
**Duration:** 5–15 minutes  
**Outcome:** New knowledge object in the knowledge engine, linked to sources, businesses, and projects

### Steps

1. Identify the knowledge to capture (insight, decision learning, research finding)
2. Copy `knowledge/_templates/knowledge-object.md`
3. Create `knowledge/domains/[domain]/[knowledge-slug].md`
4. Fill required fields: title, summary, confidence, domain
5. Add evidence items with source citations and quality ratings
6. Write applications — where and how can this be applied?
7. Set the confidence score (0.0–1.0) with written reasoning
8. Add source citations with URLs and access dates
9. Link related knowledge objects (existing IDs)
10. Link to relevant businesses and projects
11. Update `knowledge/index.md`

## Journey 11: Running Automations

**Actor:** Operator or Automation Architect agent deploying automation  
**Goal:** Create and activate an automation that reduces manual work  
**Duration:** 15–30 minutes  
**Outcome:** Automation active, tested, and logged in Automation Studio

### Steps

1. Identify the repetitive task or trigger event to automate
2. Copy `automations/_templates/automation.md`
3. Define the trigger: schedule, event, webhook, or state-change
4. Define the action: workflow, agent task, notification, or record update
5. Define any conditions (if X, then Y; unless Z)
6. Choose the platform: n8n, Zapier, or GitHub Actions
7. Build the automation in the chosen platform
8. Test with a non-production trigger
9. Activate and monitor for 24 hours
10. Log in Automation Studio with run count baseline set to 0
11. Set a review reminder for 7 days
